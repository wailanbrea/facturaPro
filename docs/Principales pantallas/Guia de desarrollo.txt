# Reglas de trabajo

## Regla 1 — No saltar fases

No se debe avanzar a una fase posterior hasta que la fase actual esté terminada, probada y marcada como completada.

## Regla 2 — Backend como fuente de verdad

El backend debe recalcular siempre:

- subtotal
- impuesto_total
- total
- balance_pendiente
- estado

No confiar en totales enviados desde web o móvil.

## Regla 3 — Plantilla única de factura

La misma plantilla HTML/CSS debe usarse para:

- Vista previa web
- Generación PDF

## Regla 4 — Android no genera PDF

Android consume la API y solicita al backend generar o descargar el PDF.

## Regla 5 — Configuración dinámica

No quemar en código:

- moneda
- impuesto
- garantía
- cuenta bancaria
- texto legal
- datos fiscales
- numeración

Todo debe venir desde base de datos/configuración.

## Regla 6 — Documentar errores

Todo error encontrado debe documentarse en este TODO antes de resolverlo.

```text

Error:

Fecha: 2026-05-22
Modulo: Backend / Android / PDF
Error: Estados visibles en ingles, moneda principal en DOP, datos demo insuficientes y Chrome headless fallando con "Missing headless user data directory".
Causa: Los codigos internos de estado se imprimian directo en UI, el seeder marcaba DOP/ITBIS como predeterminados, no habia facturas demo completas y Chrome se ejecutaba sin perfil temporal aislado.
Solucion: Se agrego traduccion de estados a espanol en web/API/Android manteniendo codigos internos estables; EUR e IVA 21% quedaron como predeterminados; se sembraron 4 facturas demo completas en EUR; Chrome PDF ahora usa --user-data-dir temporal y flags headless robustos.
Estado: Validado con php artisan test, db:seed, generacion real de FAC-DEMO-0001.pdf y build/lint/test Android.

Fecha: 2026-05-22
Modulo: Android / API Facturas
Error: El flujo de crear factura no permitia visualizar el borrador antes de guardarlo.
Causa: La vista previa existente dependia de una factura ya persistida y solo aceptaba invoice_id.
Solucion: Se agrego POST /api/invoices/preview para renderizar la plantilla oficial con los datos del formulario sin insertar registros; Android ahora tiene boton Vista previa en crear/editar factura y muestra el HTML oficial antes de guardar.
Estado: Validado con php artisan test, prueba especifica de no persistencia y build/lint/test Android.

Fecha: 2026-05-23
Modulo: Android / API Facturas
Error: En el detalle de facturas, un borrador solo podia previsualizarse como BORRADOR y no como la factura que se emitiria.
Causa: Android consumia la vista previa persistida existente, que renderiza el estado actual de la factura.
Solucion: Se agrego GET /api/invoices/{invoice}/issue-preview para renderizar una factura en formato de emision sin persistir cambios ni consumir numeracion; Android usa este endpoint cuando la factura seleccionada esta en borrador.
Estado: Validado con php artisan test, test especifico de no persistencia/contador, y gradle testDebugUnitTest/assembleDebug/lintDebug.

Fecha: 2026-05-23
Modulo: Android / Facturas PDF
Error: Despues de emitir una factura habia que generar el PDF manualmente antes de poder compartirlo; no existia accion directa para imprimir o enviar por WhatsApp.
Causa: El estado solo emitia un evento generico de compartir PDF y la UI no modelaba acciones de salida del documento.
Solucion: Se agregaron acciones de PDF tipadas (compartir, WhatsApp, imprimir); Android puede emitir, generar PDF, descargarlo y ejecutar la accion solicitada en un solo flujo. La impresion usa PrintManager y WhatsApp usa un intent explicito a WhatsApp/WhatsApp Business.
Estado: Validado con gradle testDebugUnitTest/assembleDebug/lintDebug.

Fecha: 2026-05-23
Modulo: Android / Configuracion / Impresora

Solucion: Se agregaron permisos Bluetooth en AndroidManifest, repositorio Bluetooth, DataStore local para la impresora seleccionada, ViewModel de ajustes y UI para buscar, guardar o quitar una impresora vinculada. La seleccion persiste entre sesiones.
Estado: Validado con gradle testDebugUnitTest/assembleDebug/lintDebug.

Fecha: 2026-05-23
Modulo: Android / Reportes
Error: La pantalla de reportes no seguia el nuevo diseno movil solicitado y tenia una jerarquia visual menos clara.
Causa: El Compose anterior estaba organizado como secciones genericas y no como dashboard compacto de KPIs, estados, monedas y movimientos relevantes.
Solucion: Se redisenio ReportsScreen en Compose con header compacto, filtros de fecha/moneda, KPIs 2x3, resumen por estado, distribucion por moneda y facturas relevantes usando datos reales del modelo actual.
Estado: Validado con gradle testDebugUnitTest/assembleDebug/lintDebug.

Fecha: 2026-05-23

Causa: El diseno anterior contemplaba multi-moneda como opcion operativa visible.
Solucion: Se elimino el selector de moneda y la distribucion por moneda; la pantalla conserva solo una etiqueta informativa de la moneda configurada y todos los importes salen de los datos reales del endpoint.

Fecha: 2026-05-23
Modulo: Android / Dashboard
Error: Las filas de actividad reciente no abrian la factura seleccionada y el grafico semanal mostraba barras simuladas cuando no habia facturas.
